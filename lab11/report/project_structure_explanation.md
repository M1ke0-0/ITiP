# Подробное объяснение проекта spring-lab3-notifications (Лаб. №11)

Данный проект демонстрирует работу механизмов **Inversion of Control (IoC)** и **Dependency Injection (DI)** в Spring Boot, а также основы создания REST-контроллеров.

---

## 1. Maven: pom.xml
Это конфигурационный файл системы сборки Maven. В нем описаны зависимости проекта и правила сборки.

*   **`<parent>`**: Подключает `spring-boot-starter-parent`. Это дает проекту стандартные настройки Spring Boot (версии библиотек, настройки компилятора).
*   **`<dependencies>`**:
    *   `spring-boot-starter-web`: Подключает всё необходимое для создания веб-приложений (сервер Tomcat, библиотеку Jackson для JSON и т.д.).
    *   `spring-boot-devtools`: Позволяет приложению автоматически перезапускаться при изменении кода (ускоряет разработку).
    *   `spring-boot-starter-test`: Библиотека для написания тестов.
*   **`<build>`**: Содержит `spring-boot-maven-plugin`, который позволяет упаковать приложение в исполняемый .jar файл.

---

## 2. Главный входной файл: Application.java
*   **Аннотация `@SpringBootApplication`**: Ключевая аннотация. Она включает автоконфигурацию, сканирование компонентов и позволяет запускать проект как обычное Java-приложение.
*   **Аннотация `@Import(AppConfig.class)`**: Явно подключает дополнительный класс конфигурации.
*   **Метод `main`**: Точка входа. Вызывает `SpringApplication.run`, который запускает контейнер Spring и встроенный сервер.

---

## 3. Конфигурация: AppConfig.java
*   **Аннотация `@Configuration`**: Помечает класс как источник определений бинов (объектов, которыми управляет Spring).
*   **Аннотация `@ComponentScan("org.example")`**: Указывает Spring, в каких пакетах искать классы, помеченные `@Service`, `@Component`, `@RestController`. Благодаря этому нам не нужно создавать каждый объект вручную через `new`.

---

## 4. Контроллеры (REST Controllers)

### HelloController.java
Самый простой контроллер для проверки того, что веб-сервер работает.
*   `GET /hello`: Возвращает простую строку.
*   `GET /greet?name=...`: Демонстрирует использование `@RequestParam` для получения данных из URL.
*   `GET /user-info`: Пример работы с несколькими параметрами.

### NotificationController.java
Связующее звено между пользователем и бизнес-логикой уведомлений.
*   Принимает в конструкторе `NotificationManager`. Spring сам находит этот менеджер и "внедряет" (inject) его сюда.
*   `GET /notify`: Команда отправить сообщение через **все** доступные способы доставки.
*   `GET /notify-via`: Команда отправить сообщение через **конкретный** сервис (например, только SMS).

---

## 5. Сервисы и интерфейсы (Бизнес-логика)

### Интерфейсы: MessageService.java и AdvancedMessageService.java
*   Определяют "контракт": каждый сервис уведомлений **обязан** иметь метод `sendMessage`.
*   Использование интерфейсов позволяет `NotificationManager` работать с абстрактными сервисами, не зная их деталей (принцип слабой связанности).

### Реализации сервисов:
Все они помечены аннотацией `@Service`, чтобы Spring создал их экземпляры автоматически.
*   **EmailService.java**: Имеет имя бина `"customEmail"`.
*   **SmsService.java**: Помечен `@Primary`. Если Spring попросят дать "какой-нибудь один MessageService", он выберет именно этот.
*   **PushService.java**: Обычная реализация.
*   **TelegramService.java**: Реализует расширенный интерфейс `AdvancedMessageService`.

### NotificationManager.java (Сердце лабораторной)
Этот класс управляет всеми сервисами. Он использует две мощные возможности Spring DI:
1.  **`List<MessageService>`**: Spring находит **все** классы, реализующие этот интерфейс, и собирает их в список. Это позволяет легко добавить новый тип уведомлений (например, Discord), просто создав новый класс с `@Service`, не меняя код менеджера.
2.  **`Map<String, MessageService>`**: Spring создает карту, где ключ — имя бина (например, "smsService" или "customEmail"), а значение — сам объект. Это позволяет выбирать нужный сервис по его имени в URL.

## Итог
Проект показывает, как с помощью аннотаций Spring берет на себя всю рутину по созданию объектов и связыванию их друг с другом, позволяя разработчику сосредоточиться только на написании логики.
