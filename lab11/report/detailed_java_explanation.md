# 🛡️ Полный гид по коду проекта: Spring Boot IoC/DI (Лаб. №11)

Данный документ содержит детальный разбор каждого компонента проекта для подготовки к защите лабораторной работы. Разбор структурирован по вопросам: Что делает? Зачем нужен? Как работает? Как вызывается?

---

## 🏗️ 1. Входная точка и конфигурация (Bootstrap)

### 📄 Application.java
*   **Что делает:** Это главный класс приложения, содержащий метод `main`.
*   **Зачем нужен:** Он служит "пусковым крючком". Без него Java не узнает, что это программа, а Spring Boot не инициализирует встроенный веб-сервер Tomcat.
*   **Как работает:** 
    *   Аннотация `@SpringBootApplication` запускает три механизма: автоконфигурацию, сканирование компонентов и пометку класса как конфигурационного.
    *   Метод `SpringApplication.run(Application.class, args)` создает Spring Context — область памяти, где хранятся и управляются все ваши объекты (бины).
*   **Как вызывается:** Вызывается виртуальной машиной Java (JVM) при запуске команды `mvn spring-boot:run`. Это корень всей иерархии вызовов.

### 📄 AppConfig.java
*   **Что делает:** Управляет настройками поиска бинов.
*   **Зачем нужен:** Для явного указания области сканирования компонентов (`@ComponentScan`).
*   **Как работает:** 
    *   `@Configuration` сообщает Spring, что этот класс является источником настроек.
    *   `@ComponentScan("org.example")` приказывает Spring просмотреть все файлы в пакете `org.example` и найти там аннотации `@Service`, `@RestController` и т.д.
*   **Как вызывается:** Spring считывает этот файл в момент инициализации контекста (сразу после вызова `main`).

## 🛠️ 2. Сервисы уведомлений (Слой бизнес-логики)

### 📄 MessageService.java (Интерфейс)
*   **Что делает:** Описывает абстрактный "контракт" на отправку сообщения.
*   **Зачем нужен:** Это реализация принципа инверсии зависимостей (DIP). Мы программируем "на уровне интерфейса", а не "на уровне реализации". Это позволяет легко добавлять новые виды связи (например, Telegram), не переписывая основной код.
*   **Почему так работает:** Интерфейс гарантирует, что у любого сервиса будет метод `sendMessage`.

### 📄 AdvancedMessageService.java (Интерфейс)
*   **Что делает:** Расширяет базовый интерфейс, добавляя метод получения типа сервиса.
*   **Зачем нужен:** Демонстрирует возможность иерархии интерфейсов в Spring.

### 📄 EmailService.java, SmsService.java, PushService.java, TelegramService.java
*   **Что делают:** Реализуют конкретную логику отправки (в данном случае — вывод сообщения в консоль).
*   **Зачем нужны:** Это фактические "рабочие руки" системы. Каждая реализация отвечает за свой канал связи.
*   **Специфика:**
    *   `SmsService` помечен `@Primary` — это делает его сервисом "по умолчанию". Если в коде попросить один сервис без уточнения имени, Spring даст SMS.
    *   `EmailService` имеет имя `"customEmail"`, что позволяет находить его в `Map` по ключу.
    *   `TelegramService` реализует `AdvancedMessageService`, демонстрируя расширенную функциональность (метод `getServiceType`).
*   **Как вызываются:** Вызываются из `NotificationManager`, когда тот проходит циклом по списку сервисов или ищет сервис в карте по имени.

## 🧠 3. Управление компонентами (Оркестрация)

### 📄 NotificationManager.java
*   **Что делает:** Управляет списком всех доступных сервисов уведомлений и распределяет задачи между ними.
*   **Зачем нужен:** Это центральный узел, который демонстрирует главную мощь Dependency Injection — автоматическую сборку коллекций.
*   **Как работает:** 
    *   Через конструктор и `@Autowired` он получает `List<MessageService>` (все сервисы сразу) и `Map<String, MessageService>` (сервисы по именам).
    *   Вам не нужно вручную добавлять туда новые сервисы — Spring находит их сам по аннотации `@Service`.
*   **Как вызывается:** Вызывается из `NotificationController`, когда приходит HTTP-запрос.

## 🌐 4. Обработка запросов (REST API)

### 📄 HelloController.java
*   **Что делает:** Принимает базовые HTTP-запросы пользователя.
*   **Как работает:** Аннотация `@RestController` говорит Spring, что ответы методов нужно сразу отправлять в браузер как текст или JSON.
*   **Как вызывается:** Напрямую пользователем через строку адреса в браузере (например, `localhost:8080/hello`).

### 📄 NotificationController.java
*   **Что делает:** Принимает параметры для отправки уведомлений и перенаправляет их в `NotificationManager`.
*   **Зачем нужен:** Это "лицо" системы. Он связывает внешний мир (HTTP-протокол) с внутренней логикой Java.
*   **Как работает:** Использует конструктор для получения `NotificationManager`. Когда вы заходите на `/notify`, он вызывает `notificationManager.notify()`.

## 📦 5. Проектный файл

### 📄 pom.xml
*   **Что делает:** Это "паспорт" проекта Maven.
*   **Зачем нужен:** В нем прописаны все библиотеки (Spring Boot Starters), версия Java и правила сборки. Без него Maven не сможет скачать нужные зависимости из интернета.
